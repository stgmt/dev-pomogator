# MCP-rails — следующая большая волна: глубокий анализ (2026-06-07)

**Запрос юзера:** «делай живой генератор; запретить grep по спекам — всё через MCP only,
централизованно, чтоб мы контролировали что агент делает, и логи; создание спеки через MCP
вызовом claude -p/-bg агентов — агент на каждый этап + агент-оркестратор-проверятор;
это следующая большая волна рефактора легаси-подхода». Размещение (выбор юзера): всё в
spec-generator-v4 — FR-39/40/41 + Phase 17.

## Зачем (мотивация — не «запрет», а контроль + лог)

Сегодня агент работает со спеками как с обычными файлами: Read/Grep/Edit врассыпную,
31 скилл ссылается на прямые пути, никакого следа «кто что прочитал/изменил и зачем».
Централизация через MCP даёт ровно то, что просит юзер: **одна дверь → один лог →
управляемое поведение агента** + живая валидация на записи (генератор перестаёт писать
вслепую). Запрет грепа — следствие, не цель.

## Хребет: граница агент vs движок (и примирение с FR-21)

Без этой границы спека самоблокирующаяся. **Запрет распространяется на TOOL-CALLS
АГЕНТА** (Read / Grep / Glob / Edit / Write / Bash-cat по `.specs/**`) — и ТОЛЬКО на них.
**Движок** — builder, парсеры, CLI (spec-verdict, corpus-health, spec-status,
validate/audit), хуки, резолверы spec-backlog и сам MCP-сервер — читает и пишет диск
in-process как раньше: он и ЕСТЬ бэкенд этой двери. Поэтому противоречия с FR-21
(«CLI обязан работать без MCP-сервера») НЕТ: FR-21 — про деградацию ДВИЖКА, FR-39 —
про дисциплину АГЕНТА. Оба истинны одновременно. (Зафиксировано первой строкой FR-39;
правило verify-divergent-contracts соблюдено — обе стороны прочитаны и примирены.)

## Чего не хватает (проверено живыми пробами)

- **Чтение**: `get_node` УЖЕ отдаёт `body` (проверено: keys = id,type,title,file,line,
  anchors,body,spec). Не хватает чтения ЦЕЛЬНЫХ документов (проза вне узлов: README,
  DESIGN-абзацы, RESUME) → нужен `read_spec_doc` (+ лог доступа).
- **Запись**: MCP сегодня read-only (14 тулзов). Нужна мутирующая поверхность с живой
  валидацией — само сердце «живого генератора».
- **Лог**: нет следа агентских обращений → `.dev-pomogator/logs/spec-access.jsonl`
  (append-only, O_APPEND): `{ts, tool, args-digest, caller, decision}`.

## Жёсткая цепочка зависимостей (enforce — СТРОГО последним)

Включить deny-хук раньше готовности двери = окирпичить и себя, и все 31 скилл в первый
же день (этой сессией я сам десятки раз делал Read по `.specs/`). Порядок:

1. **P17-1 read-sufficiency** — `read_spec_doc` (+ опц. `list_spec_docs`); каждый
   read-вызов логируется. Гейт: авторинг-ревью реально проходим MCP-only (проба).
2. **P17-2 mutation surface** — «живой генератор»: `propose_spec_change` /
   `apply_spec_change` (или create_spec/write_spec_section): сервер прогоняет
   conformance + анкеры + form-контракты ДО записи; отказ = findings list, агент
   правит и повторяет. Запись атомарна, логируется. Оборачивает резолверы/scaffold.
3. **P17-3 shadow-хук** — PreToolUse на Read|Grep|Glob|Edit|Write|Bash: матч `.specs/**`
   → ЛОГ нарушения (без блока). Метрика: violations/день → 0 по мере миграции скиллов.
4. **P17-5 миграция корзины 1** — инструкции скиллов переписываются на MCP-вызовы.
5. **P17-6 enforce** — флип shadow→deny (env `SPEC_ACCESS_ENFORCE`), escape hatch
   `[skip-spec-access: <reason≥8>]` + JSONL-аудит по образцу scope-gate. Гейт флипа:
   shadow-лог чистый N дней + P17-1/2 доказаны.
6. **P17-7/8 агенты по фазам + оркестратор-проверятор** — параллельная дорожка,
   зависит от P17-1/2 (агентам нужна полная MCP-дверь):
   - На каждый этап create-spec — свой headless-агент (`claude -p`, для длинных фаз
     `-bg`-паттерн из claude-fallback): discovery-agent, requirements-agent,
     finalization-agent, audit-agent. **MCP-only принуждается через allowed-tools**
     (выдаём MCP-тулзы, НЕ выдаём Read/Grep по спекам) — это второй слой enforcement,
     чище хука; хук остаётся для интерактивной главной сессии.
   - Оркестратор-проверятор: спавнит фазу → между фазами гоняет spec-verdict +
     get_spec_status → RED = вернуть фазу агенту с gap list, не пускать дальше.
     Определения агентов: `.claude/agents/spec-phase-*.md` (директории сейчас нет —
     создаётся в P17-7).

Анти-грабли из собственной истории (обязательные для P17-3/6): новый хук —
**зарегистрирован живым в обоих манифестах + deps-absent прогон + пин в SPECGEN004_52 +
самозащита в meta-guard** (на этой неделе оживляли пять мёртвых стражей — не отгружаем
шестой мёртвым).

## Риски

| Риск | Митигация |
|---|---|
| Окирпичивание авторинга ранним enforce | цепочка выше; enforce последний; shadow-метрика как гейт |
| Производительность: каждый Read-хук на любой файл | хук фильтрует по `.specs/` ДО любого I/O (паттерн form-guards-dispatch) |
| Латентность фазовых агентов (`claude -p` минуты) | -bg паттерн + оркестратор ждёт асинхронно; фазы и так длинные у человека |
| Mutation API становится вторым валидатором | НЕ дублировать: сервер вызывает СУЩЕСТВУЮЩИЕ conformance/anchors/form-парсеры in-process |
| Лог пухнет | ротация по образцу audit-logger (10MB/30дн) |

## Definition of victory волны

Агент в интерактивной сессии и все фазовые агенты читают и пишут спеки ТОЛЬКО через MCP;
каждый доступ в логе; запись валидируется до касания диска; create-spec
управляется оркестратором с проверятором между фазами; deny-хук живой и не мешает
движку. validate/verdict/corpus-health работают как раньше (FR-21 цел).
