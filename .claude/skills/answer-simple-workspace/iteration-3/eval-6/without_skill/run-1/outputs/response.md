## Общая структура волн

Разбивка на 3 волны по 2 недели выглядит разумно:

- **Волна 1 (payments, billing, invoicing)** — high QPS первыми, чтобы раньше поймать сюрпризы под нагрузкой. Логично.
- **Волна 2 (analytics, reports, dashboards)** — read-heavy сервисы, downtime для них менее болезненный для пользователей.
- **Волна 3 (notifications, audit, scheduler, search, recommendations, support)** — остальные.

Итого ~6 недель на полную миграцию — реалистично, если параллельно нет других крупных релизов.

## Что я бы уточнил

1. **Почему high-QPS первыми?** Обычно делают наоборот — начинают с low-risk сервисов чтобы откатать процедуру, потом идут к critical path. Если payments упадёт первым — blast radius максимальный. Есть ли причина начинать именно с них (например, они блокируют что-то для остальных)?

2. **30 минут downtime на сервис** — это hard requirement новой gates системы или можно zero-downtime? Для payments/billing 30 минут — это серьёзно, обычно для них нужен blue-green или canary.

3. **24 часа staging soak** — хорошо, но что именно мониторим? Нужны конкретные SLO/метрики (error rate, latency p99, throughput), иначе "прошло 24 часа без падений" ≠ готово к prod.

4. **Rollback playbook** — на каком уровне? Per-service? Per-wave? Если волна 1 закончилась, потом в волне 2 нашли проблему общую для gates системы — откатываем все 6 сервисов или только последний?

5. **SRE согласование окон** — это блокер для старта? Стоит запустить параллельно с подготовкой playbook, чтобы не ждать.

## Что бы добавил

- **Pre-migration checklist** на каждый сервис: backup БД, health checks, alerting проверен, on-call расписание.
- **Communication plan**: кого уведомляем за 24ч/1ч/при старте/при завершении (особенно для payments — там и финансовые партнёры).
- **Success criteria для перехода между волнами**: что должно быть зелёным после волны 1, чтобы запускать волну 2.
- **Buffer week** между волнами на случай непредвиденного — 6 недель подряд без зазора рисково.
