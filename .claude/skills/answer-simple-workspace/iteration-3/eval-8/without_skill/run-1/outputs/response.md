## По текущему коду

Функция `validateConfig` сейчас возвращает `ValidationResult`, но `ajv.validate()` по умолчанию возвращает `boolean` — а ошибки лежат в `ajv.errors`. Если `ValidationResult` — это твой кастомный тип, то внутри функции должна быть упаковка типа:

```typescript
function validateConfig(cfg: ConfigSchema): ValidationResult {
  const valid = ajv.validate(schemaV2, cfg);
  return { valid, errors: valid ? [] : (ajv.errors ?? []) };
}
```

Иначе TypeScript должен ругаться на возврат `boolean` вместо `ValidationResult`. Проверь, что компилируется без ошибок.

Второй момент — `ajv.validate()` мутирует состояние инстанса (`ajv.errors`). Если `ajv` шарится между запросами или потоками, две параллельные валидации затрут ошибки друг друга. Безопаснее скомпилировать схему один раз:

```typescript
const validate = ajv.compile(schemaV2);
// потом в функции:
const valid = validate(cfg);
return { valid, errors: valid ? [] : (validate.errors ?? []) };
```

`validate.errors` локальны для скомпилированной функции, гонок не будет.

## По следующему шагу — error reporting

Прежде чем кодить, определись по нескольким развилкам:

1. **Формат ошибок наружу** — сырые `ErrorObject[]` от AJV (с `instancePath`, `schemaPath`, `keyword`, `params`) или человекочитаемый текст? Для CLI обычно нужен текст, для API — структурированный объект.
2. **Где репортить** — `console.error` / `throw` / возврат в `Result`-типе / отдельный логгер? Это сильно зависит от того, как `loader.ts` сейчас обрабатывает ошибки в других местах. Стоит посмотреть.
3. **Локализация путей** — `instancePath` у AJV выглядит как `/database/host`. Часто хочется превратить в `database.host` и подсветить, какое значение пришло.
4. **Fail-fast или собрать всё** — `ajv.compile({ allErrors: true })` соберёт все ошибки за один проход, иначе остановится на первой.
