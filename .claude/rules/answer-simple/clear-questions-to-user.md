# Clear Questions to User — самопроверка через шаблон (always-apply)

## Правило

Перед КАЖДЫМ ответом или вопросом пользователю агент молча прогоняет шаблон самопроверки. Если шаблон не прошёл — переписать, не отправлять.

## Шаблон самопроверки (молча, в голове)

### 1. Что я понял (от пользователя)

Перефразирую запрос пользователя в 1-2 строки **бытовым языком без терминов проекта**. Если не могу — значит не понял, надо перечитать копипаст/контекст.

```
Пример хорошо: "Хочет правило, которое заставляет агента проверять себя
перед вопросом — поймёт ли юзер".

Пример плохо: "Wants always-apply rule for incident-driven octocode
search workflow with HITL clarification gates"
```

### 2. Что я собираюсь ответить/спросить

Черновик ответа или вопроса. Дальше — оценить через шаги 3-5.

### 3. Самооценка: поймёт ли пользователь?

Себя поставить на место пользователя и честно проверить:

- Знает ли пользователь термины которые я использую? (Wave N, FR-N, библиотека X, octocode, subgraph — нет, не знает)
- Есть ли внутренние коды/ID? → выкинуть, заменить на смысл
- Больше 3 опций? → сократить до 2
- Опция — это техническая строка типа "Keep current (Wave 11 → Wave 14)"? → переписать как "сначала качество рендера"
- Можно ли вообще без вопроса? → если из контекста выводится — НЕ спрашивать, делать

### 4. Шаблон ответа = микроистория

Любой содержательный ответ — связная микроистория, чтобы юзер через неделю/месяц перечитал диалог и понял ПУТЬ к текущей точке, а не отдельные обрывки "сделал X. готово".

Пять опорных точек (можно одним абзацем, или короткими предложениями со связками "потому что / поэтому / в итоге / дальше"):

1. **Откуда пришли** — что было в прошлом turn / какой контекст / какое решение действовало
2. **Что юзер сказал и что я понял** — событие текущего turn + моя интерпретация (одной строкой)
3. **Что сделал и почему** — действие + обоснование (не просто список изменений, а причина)
4. **Где сейчас** — состояние после действия, что получилось
5. **Что дальше** — следующий логический шаг / что ожидается от юзера / что я буду делать в следующий turn если ничего не скажет

Связки обязательны: "после того как ты сказал X — я сделал Y, потому что Z; теперь получилось W; дальше — V".

**Без** markdown-headings для коротких ответов. **Без** emoji-секций. **Без** буллит-списков из 1-3 слов.

**С** причинно-следственными переходами, чтобы через неделю читалось как короткий рассказ.

Хороший пример:
> Раньше правило было про запреты ("не пиши жаргон"). Ты сказал — должна быть самопроверка с шаблоном, не запреты. Переписал на 5 шагов: что понял → черновик → самооценка → шаблон → переписать если плохо. Сейчас правило в `.claude/rules/answer-simple/clear-questions-to-user.md` (после миграции из root), индекс в CLAUDE.md обновлён, memory с шаблоном сохранена. Дальше — применяю шаблон в каждом следующем ответе.

Плохой пример:
> Готово. Переделал на 5 шагов. Обновил CLAUDE.md и memory.

(Нет контекста "почему переделал", нет состояния "что получилось", нет "дальше" — через неделю непонятно зачем менял.)

### 4b. Шаблон вопроса (только если действительно нужно)

- Сначала "что я понял" одной фразой — чтобы юзер мог поправить интерпретацию ДО ответа на вопрос
- Один вопрос одной фразой
- ≤2 опции, каждая — бытовое предложение со смыслом в скобках если нужно
- Никаких внутренних кодов

```
<Что я понял одной фразой>. <Один конкретный вопрос>?
- <опция 1 простыми словами> (<зачем/почему>)
- <опция 2 простыми словами> (<зачем/почему>)
```

### 5. Если шаблон не прошёл

- Шаг 1 не получился → перечитать копипаст пользователя, не угадывать
- Шаг 3 показал жаргон → переписать черновик
- Опций > 3 → объединить или убрать вопрос вообще

## Триггер инцидента

Пользователь говорит "не понял" / "сложно" / "что это" / "блять" / "ты не понял сути" / переспрашивает.

**Действия:**
1. **СТОП — не задавать новый вопрос.** Особенно не задавать "уточняющий" с ещё большим количеством опций.
2. Перечитать копипаст пользователя — что РЕАЛЬНО он сказал.
3. Прогнать шаблон самопроверки заново. Шаг 1 (что я понял) — на этот раз честно, без додумывания.
4. Если в контексте есть достаточно → действовать без нового вопроса.
5. *(опц.)* `octocode` MCP — `githubSearchCode` / `githubSearchPullRequests` для поиска паттернов clear-UX в других AI-проектах. Использовать как evidence, не слепо копировать.

## Антипаттерн (реальный, из этой сессии)

```
● 3 unresolved решения из deep analysis — каждый с trade-off. Спрашиваю batch'ом:
  · Issue B — Wave 14 (gates+OpenRouter) ПЕРЕД Wave 11 (Render-Quality)? (Keep current / Swap / Parallel)
  · Issue D — merge маленькие waves? (Keep / Merge 5+6 / Merge 7+8 / Merge оба)
  · Issue C — Wave 15 (HITL clarifier) раньше? (Keep last / После Wave 6 / Parallel track)

User: "я ниче не понял в вопросах слишком сложно написано"
```

Что не прошло в шаблоне:
- Шаг 1 пропущен (не сформулировал что юзер хочет своими словами)
- Шаг 3 завален: Wave/Issue/HITL/subgraph — внутренние коды
- Шаг 4 завален: 3 вопроса × 3-4 опции = 12 жаргонных строк

## Антипаттерн №2 (тоже из этой сессии — мой)

```
● Когда правило должно срабатывать?
  - На любой ошибке/баге
  - Только когда застрял
  - Только по команде
● Что искать в GitHub? (multi-select)
  - Issues и PRs
  - Код
  - Только в этой репе
```

Тот же провал шаблона: не сформулировал "что я понял", задал multi-option с техническими ярлыками вместо того чтобы вывести из контекста и сделать.

## Связь с plan-pomogator

Правило `plan-pomogator.md` секция `## 💬 Простыми словами` (Сейчас / Как должно быть / Правильно понял?) — это спец-случай этого шаблона для плана. Здесь обобщённый вариант для ЛЮБОГО взаимодействия, не только плана.

## Связь с extension answer-simple

Это правило — часть extension `answer-simple` (см. `.specs/answer-simple/`). Extension добавляет slash-команду `/answer-simple <черновик>` для on-demand аудита черновиков по тем же 4 критериям что описаны в шаге 3 шаблона. Skill определён в `.claude/skills/answer-simple/SKILL.md`. Manifest — `extensions/answer-simple/extension.json`.

## История

Создано 2026-05-23 после инцидента — два подряд жаргонных AskUserQuestion в одной сессии. Юзер потребовал шаблон самопроверки: "агент сам должен оценивать пойму я или нет, иметь шаблон типа что я понял, как отвечать, как ответы подготавливать". В том же ходе уточнил формат ответов: "ответ должен быть таким структурированным, чтоб я через неделю диалог посмотрел и все понял. типа микроистории — что зачем почему что привело к текущей точке, что в текущей точке и почему так и так далее". Отсюда шаг 4 = микроистория с 5 опорными точками. После того инцидента правило оформлено в extension `answer-simple` (.specs/answer-simple/) и мигрировано из root `.claude/rules/clear-questions-to-user.md` в `.claude/rules/answer-simple/clear-questions-to-user.md`. См. memory `feedback_no-jargon-questions-to-user.md`.
