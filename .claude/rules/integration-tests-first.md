# Integration Tests First

## Правило

Тесты ОБЯЗАНЫ быть интеграционными — проверять реальный end-to-end flow через production код. Unit тесты допустимы как дополнение, но НЕ как замена интеграционных.

## Почему

Unit тесты функции `resolveClaudeStatusLine()` проходили 100%, но баг "ccstatusline не устанавливается" жил месяцами — потому что ни один тест не проверял реальный flow: installer → читает extension.json → вызывает функцию → пишет в settings.json.

## Что считается интеграционным

- Вызывает `runInstaller()` и проверяет результат на файловой системе
- Запускает реальный скрипт через `spawnSync`/`execSync` и проверяет stdout/exit code
- Читает реальные файлы после выполнения действия

## Что НЕ считается интеграционным

- Импортирует функцию и вызывает с захардкоженными аргументами
- Мокает входные данные вместо чтения из реальных конфигов
- Проверяет логику в изоляции от файловой системы и child processes

## При обнаружении

Если найден тест который проверяет критический flow только через unit:

1. Пометить `UNIT-ONLY: {описание}` — кандидат на интеграционный тест
2. Создать интеграционный тест рядом (тот же describe, новый it с суффиксом `(integration)`)
3. Unit тест оставить как есть — он не мешает

## Чеклист при написании тестов

- [ ] Каждый FR покрыт минимум 1 интеграционным тестом (runInstaller / spawnSync)
- [ ] Unit тесты НЕ считаются покрытием FR без парного интеграционного
- [ ] Новый тест проверяет реальный файл/процесс, не mock/stub
