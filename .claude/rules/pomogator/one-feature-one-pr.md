# One feature, one PR — против over-split

## Правило

**1 фича = 1 PR в идеале.** Не дробить связную фичу на «foundation / part 2 / part 3 / benchmark» если каждый кусок **не имеет самостоятельной ценности** без следующих.

Конкретные шаги плана работ — это work breakdown, **не** PR boundaries. План декомпозирует задачу для исполнителя, а не для ревьюера.

## Когда НЕЛЬЗЯ дробить (объединять в один PR)

- Слои одного компонента: типы + парсер + builder + бенчмарк (все нужны вместе чтобы фича работала).
- Имплементация + её тесты — всегда в одном PR.
- «Foundation» PR без самостоятельной ценности (только types, без потребителя типов).
- Куски, каждый из которых проверяется только в контексте следующего.
- Конкретные шаги плана (Phase N step M в TASKS.md) — план разбивает работу для исполнения, а не PR boundaries.

## Когда дробить ОПРАВДАНО

- **Независимые фичи** — разные подсистемы / разные slug в `.specs/`.
- **Mechanical setup ≠ feature impl** — например, добавление зависимостей в `package.json` + первый коммит фичи на бэке = отдельные PR, если setup нужен и другим фичам.
- **Ship-then-iterate** — фича должна реально работать в проде, и только потом строится следующая поверх (true sequential dependency).
- **Diff действительно большой** — code logic > ~1500 LOC AND есть clean cut on responsibility boundary (не «artificial cut» по «удобству»).

## Self-check перед `gh pr create`

- [ ] Может ли ревьюер понять PR за один заход **без чтения предшествующих**?
- [ ] Если убрать этот PR — остаётся ли в следующем работающая фича? Если **ДА** → можно отдельно; если **НЕТ** → объединять.
- [ ] PR описывается одним предложением «делает X»? А не «foundation для X, который доделаем в PR-N»?
- [ ] CI и code review занимают ~3 мин + N минут review. Оправдан ли overhead × M раз для размера фичи?

## Антипаттерн (реальный, из этой сессии)

План разложил Phase 1 v4 SpecGraph на 5 sub-PR: foundation → gherkin → ndjson → builder → benchmark. Я начал делать буквально по плану — отправил PR #31 «foundation» (типы + один парсер из четырёх). User pushback: «почему каждый микро-чендж в отдельном PR? откуда такие правила?».

**Что должен был сделать:** одной веткой собрать всю Phase 1 (≈ 4-5 дней работы, ~3000 LOC) — типы + 3 парсера + builder + incremental rebuild + conformance checker + cold-start benchmark — отправить **один** PR «Phase 1: SpecGraph builder + parsers + conformance». Diff больше, но один проход CI + ревью вместо пяти, и ревьюер видит фичу целиком.

**Tell для over-split'а:** PR description содержит фразы «follow-up sub-PR доделает X» / «foundation для следующего PR» / «фаза N будет в N следующих PR-ов».

## Когда план сам подсказывает over-split

Если детальный план (`.claude/plans/<slug>.md`) разбивает одну фазу на 3-5 sub-PR — это **подсказка** для исполнителя, а не обязательство. На моменте `gh pr create` спросить себя:
- Эти sub-PR имеют самостоятельную ценность?
- Сможет ли каждый merge'нуться, не дождавшись следующего, и репо останется в рабочем состоянии?

Если хотя бы один «sub-PR» нет — объединять в один.
