# Verify Status Against Code Before Acting — Status Docs Drift

## Правило

**Прежде чем планировать или делать работу на основании статуса из документа
(спека, TASKS.md, README, FR.md, план, правило) — verify этот статус против
КОДА: `ls` файла + `grep` символа + запуск теста. Статусы в доках дрейфуют от
реальности кода. "Помечено TODO/`[ ]`/deferred/planned" ≠ "не сделано".
"Правило говорит X живёт" ≠ "X живёт".**

Default: доку про статус не доверяй — проверь за 30 секунд.

## Почему

Status-доки отстают от кода систематически. `npm run check:status-drift` на
этом репо сразу нашёл **30 строк**, где TASKS.md помечает `[ ] Создать
tests/e2e/X.test.ts`, а файл `tests/e2e/X.test.ts` **уже существует** — задача
не отмечена галочкой, хотя код есть (либо сделано и не закрыто, либо заглушка).

Действовать по устаревшему статусу = жечь циклы: делаешь уже готовое, чинишь
несуществующий баг, или ломаешь то что трогать не надо (неполный список).

## Cross-project evidence (sister repo, 1 сессия — 6 раз)

В `presentation-reels` (родственный репо) reproduce 6 раз ловил расхождение:
правило "shim ещё живёт" → удалён 2 волны назад; спека "FR-29 DEFERRED" →
adapter + 30 тестов проходят; "outputs пустые = баг" → артефакт запроса;
"флачит сервер" → timeout < runtime; правило "4 consumers" → реально 6;
"баг персистенса" → CLI просто не показывал данные. Все 6 пойманы проверкой
кода, не доверием доку.

## Когда применять (mandatory)

- Перед планированием работы из спеки/TASKS.md/README/FR.md/плана/правила с
  маркером `[ ]`/TODO/deferred/planned/🟡
- Перед "починкой" заявленного бага — сначала reproduce что баг реален
- Перед удалением/рефактором "legacy" — `grep` ВСЕ consumers, не доверяй
  списку из дока (он неполный)

## Mandatory check (30 секунд)

```bash
ls <claimed-path>                      # "не создан" но exists → drift
grep -rln "<symbol>" tools/ plugin-dev/ tests/   # используется → возможно done
npm run test:bdd -- --name "<scenario>"          # "тестов нет" → запусти
```

## Автоматизация — `check:status-drift`

`scripts/check-status-drift.mts` (`npm run check:status-drift`): сканит
`.specs/` + `.claude/rules/` на маркеры (`[ ]` контекст / TODO / DEFERRED /
planned / 🟡 / 🔴) рядом с код-путём (`tools/`, `tests/`, `plugin-dev/`,
`scripts/`). Если путь СУЩЕСТВУЕТ → флаг "помечено incomplete, но код есть →
verify". Conservative ("pending" исключён — матчит имена фич). Не доказательство
— существование файла ≠ фича готова; флаг → проверь grep+тест.

## Anti-patterns

| ❌ | ✅ |
|---|---|
| "TASKS.md: `[ ] создать X` → создаю" | `ls X` сначала — может уже быть |
| "Правило: N consumers" → трогаю N | `grep -rln` ВСЕ → может быть N+2 |
| "Баг X" → чинить | reproduce X сначала |
| Читать статус из `.specs/backlog/` как актуальный | verify против кода |
