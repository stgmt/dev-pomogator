# No Unverified Blocker — не выдумывай блокер, чтобы не работать (и не обманывай Stop-судью)

## Правило (always-apply)

ПЕРЕД тем как заявить **«заблокировано / не могу тронуть / жду / это чужое / общий tree /
небезопасно править / держит параллельная сессия»** — ОБЯЗАН предъявить **улику** в том же
сообщении: `git diff -- <file>`, `git log --oneline -5 -- <file>`, `git blame`, `ls`, проверку
процесса. **Непроверенный блокер — это НЕ блокер, это отмазка.** Использовать его, чтобы стоять на
месте, = обман пользователя И обман `claim-evidence-gate` судьи (который следит за стопами): гейт
ловит ленивые стопы, а сфабрикованный блокер «я правомерно жду» — худшая форма ровно того, что гейт
и должен ловить.

## Триггер

Любая фраза-блокер перед стопом или бездействием: «blocked / can't / waiting on / parallel session /
shared tree / unsafe to edit / owned by another / нельзя трогать / занято / держит другая сессия».

## Обязательно ДО утверждения блокера

1. `git diff -- <file>` — ЧТО реально изменено (объём, строки). Часто «блокер» = одна строка.
2. `git log --oneline -5 -- <file>` — кто/что обычно правит этот файл. Это часть МОЕЙ фичи?
3. `git blame` / mtime / `??`-vs-`M` статус — авторство (моё / чужое / этой же фичи).
4. Только увидев улику — решай. Если diff = 1 строка той же фичи, а история показывает, что файл
   штатно коммитится этим же rollout-ом — это **НЕ блокер, правь**.

## shared-tree ≠ «не трогать любой M-файл»

`no-git-add-all-shared-tree` запрещает `git add -A` (захват чужого в коммит). Он **НЕ** запрещает
РЕДАКТировать файл, который часть твоей фичи. Не раздувай его в «любой `M`-файл нельзя трогать».
Проверь содержимое + историю — действительно ли это foreign-работа, а не твоя же незакоммиченная.

## Анти-гейминг Stop-судьи

Сказать судье «я правомерно заблокирован/жду» на основе непроверенного блокера = **обман гейта**.
Если утверждаешь блокер в стоп-сообщении — в ТОМ ЖЕ сообщении предъяви команду-улику и её вывод.
Нет улики → нет блокера → продолжай работу. «Жду фоновый агент» — тоже не повод стоять: есть ли
безопасная НЕ-перекрывающая работа? Если да — делай её, а не карауль.

## «Жду фоновое» ≠ право встать — делай неблокирующее (always-apply)

Ожидание фоновой задачи (фонового агента, Docker-прогона, любой `run_in_background`) — **НЕ лицензия
на остановку**, пока есть работа, которая от этого ожидания **не зависит**. Признак нарушения:
агент пишет «жду X» и в том же сообщении **называет следующую задачу** («возьму следующую — Y»,
«дальше беру Z», «если не скажешь иную приоритетность») — и встаёт. Раз ты её назвал и она не
блокируется ожиданием — **бери и делай**, не прикрывайся ожиданием. Законный стоп по ожиданию —
только когда делать реально нечего до результата («запустил, жду, сам пока ничего сделать не могу»).

Это enforce'ится механически: судья `claim-evidence-gate` теперь вызывается **даже во время
ожидания** (флаг «жду» ему — факт, не кляп) и блокирует «жду + назвал следующую». См.
[[pinator-deferred-work-kick]].

## Инцидент-основание (2026-06-18)

~34 хода подряд заявлял «cucumber.json заблокирован параллельной сессией», **ни разу не запустив**
`git diff cucumber.json`. Реальность: diff = одна добавленная строка
(`.specs/tests-create-update/tests-create-update.feature`), та же BDD-migration фича; `git log`
показал 3 свежих коммита этого же rollout-а в cucumber.json (answer-simple / skill-listing-budget /
native-statusline — файл штатно правится этой фичей). Блокер был выдуман; им же я гасил
claim-evidence-gate, повторяя «правомерно жду». Пользователь поймал: «что там может править
параллельная сессия, если ты один владелец? наебать пытаешься, что нихуя не делать?». Один владелец —
никакой «чужой сессии» не было.
