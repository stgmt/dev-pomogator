# Claim-Evidence Gate — Functional Requirements

## Контекст

Stop-хук, который ловит инцидент-первопричину этой фичи: агент презентует РЕЗУЛЬТАТ (таблицу вердиктов fact-check, «работает», «не существует», `[VERIFIED via X]`), которого в этом ходе не производил — вообразил вывод инструмента вместо запуска. Существующие сторожа (pinator) ловят формулировку по словам-триггерам, а не наличие улики; tester/researcher вообще без хука. Этот гейт — объединение: один сторож на класс «заявил результат без улики».

Реализация: `tools/claim-evidence-gate/` (`turn_window.ts`, `claim_classifier.ts`, `claim_evidence_gate_stop.ts`). Зарегистрирован в `.claude-plugin/hooks.json` (canonical) + `.claude/settings.json` (dogfood).

## FR (Functional Requirements)

- **FR-1**: Хук SHALL извлекать turn-window — всё после последнего реального user-сообщения (не tool_result): последний assistant-текст (claim) + все `tool_use {name,input}` главной цепочки, ИСКЛЮЧАЯ sidechain-линии.
- **FR-2**: Классификатор SHALL распознавать 4 класса заявлений на `stripCode(text)`: `analysis-verdict` (≥2 строки вердикт-токенов PASS/FAIL/✅/❌), `works-done` («работает/починено/фикс деплоен/тесты зелёные», standalone, не negated, не explainer), `not-found-impossible` («не нашёл/не существует/архитектурно невозможно»), `verified-marker` (`[VERIFIED via X]` на raw-тексте).
- **FR-3**: Гейт SHALL блокировать первый класс, чья улика отсутствует: analysis-verdict/works-done → ≥1 исполнитель (Bash/PowerShell/Task/Agent/mcp__*); not-found → ≥`CLAIM_GATE_MIN_SEARCH` (default 2) поисковых вызовов (Grep/Glob/WebSearch/WebFetch/octocode/Task); verified-marker → tool_use, чьё имя/вход содержит токен из X.
- **FR-4**: `CLAIM_GATE_ENABLED` SHALL принимать `true` (enforce, **дефолт**), `shadow` (только лог, никогда не блок), `false` (выкл).
- **FR-5**: Каждое срабатывание SHALL дописываться в `.dev-pomogator/.claim-evidence-gate-fires.jsonl` (ts, class, need, tool_uses, claim_snippet, mode, session_id, cwd) — даже в shadow.
- **FR-6**: Гейт SHALL иметь anti-loop (одинаковый claim-hash → approve; > `CLAIM_GATE_MAX_RETRIES` за cooldown → approve), self-marker short-circuit, honor `stop_hook_active`, и fail-open на любой ошибке (вернуть `{}`).
- **FR-7**: Гейт НЕ ДОЛЖЕН блокировать репорт результата, полученного ПОСЛЕ реального запуска в том же user-ходе (улика в окне) — окно ограничено сообщениями юзера, не ходами ассистента.

### Реализация-2: судить РЕАЛЬНОСТЬ, а не нарратив (решения владельца 2026-06-18)

Контекст: судья-слой (FR-49 в `spec-generator-v4`) судил по `{finalMessage, tools, openTasks}` — два из трёх пишет сам агент или они вводят в заблуждение (`openTasks` = глобальный бэклог репы). Это давало жор токенов и геймабельность (Goodhart: гейт награждает того, кто лучше пишет обещания). Анализ: `audit-reports/pinator-token-burn-analysis.md`. FR-8..FR-13 — валидированные владельцем решения, ЗАПИСАНЫ ДО реализации.

- **FR-8** (решение 1 = 1b): Промпт судьи (FR-49e, `meridian-judge.ts::buildJudgePrompt`) SHALL быть переписан так, что НАБЛЮДАЕМЫЕ tool-факты ПЕРВИЧНЫ в вердикте; формулировка «running tools this turn is IRRELEVANT» SHALL быть снята. Текст агента — вторичный tiebreaker. Осознанный трейд-офф: LLM-судья остаётся (дешевле полной перестройки), поэтому частичная текст-зависимость сохраняется — её добивают FR-9..FR-11.
- **FR-9** (решение 2 = 2a + C1): Предусловие «осталась незакрытая работа» SHALL вычисляться по SCOPE СЕССИИ, а не по глобальному бэклогу репы. Scope SHALL выводиться из транскрипта: пути файлов из `tool_use` Edit/Write/apply_spec_change ЭТОЙ сессии → spec-папки → open-задачи ТОЛЬКО в них. ОДИН источник SHALL использоваться И для входа судьи (`openTasks`), И для per-prompt баннера. Обязательное следствие: сессия, не редактировавшая ни одной спеки, → scope = 0 → гейт НЕ взводится.
- **FR-10** (решение 2 = B2): Вход судьи SHALL включать наблюдаемые, агент-независимые факты, собранные ХУКОМ: число мутирующих `tool_use` в ЭТОМ ходе (Edit/Write/Bash/apply_spec_change); менялись ли файлы рабочего дерева в этом ходе (`git diff --stat` / хэш `git status --porcelain`); есть ли pending background task. Вердикт SHALL приниматься по этим фактам в первую очередь (см. FR-8). Где git недоступен (Docker без `.git`) — факт fail-open (не блок).
- **FR-11** (решение 3 = гибрид со скорингом): Гейт SHALL детектить отсутствие прогресса между киками: если между двумя заблокированными стопами было 0 мутирующих `tool_use` И 0 изменений рабочего дерева → агент не движется → release (approve). Сигналы (tool-delta, tree-delta, повтор claim, верификация блокера) SHALL комбинироваться СКОРИНГОМ (не один булев флаг) в решение block/approve/release. Прошлое утверждение агента НЕ есть улика; заявленный блокер SHALL требовать наблюдаемого пруфа — хук САМ гонит `git diff/log` по названному файлу — иначе дефолт «не заблокирован». Прямой фикс парот-цикла: судья не повторяет за агентом, агент не повторяет за собой.
- **FR-12** (решение 4) — **DEFERRED / СЛОЖНОЕ РЕШЕНИЕ (НЕ на первый проход)**: APPROVE-кейс «агент легитимно завершает ход, ожидая делегированную фоновую задачу» (`run_in_background`-spawn без completion). Помечено владельцем как сложное/неопределённое: надёжное сопоставление spawn↔completion по транскрипту + взаимодействие с монитором долгих тулов (FR-13) ещё не спроектированы. НЕ планируется в первую итерацию.
- **FR-13** (новый кейс владельца 2026-06-18 — монитор долгого тула + эскалация к человеку): Когда активен долгоиграющий инструмент (Monitor ИЛИ фоновый Bash/тесты), хук SHALL читать его ВЫВОД с диска (persistent-логи: `.dev-pomogator/.docker-status/test-run-*.log`, `.bg-logs/*`), хранить в маркере последние N (≈5) хвостов/хэшей и сравнивать их. «Прогресс» SHALL определяться по выводу тула ИЛИ по наблюдаемым фактам разговора (tool-delta/tree-delta из FR-10/FR-11). IF вывод не менялся ≥5 итераций AND нет прогресса в разговоре THEN состояние «завис/цикл» → хук SHALL вернуть block с инструкцией агенту вызвать AskUserQuestion (ждать / убить / продолжить). Механизм эскалации (выбран по умолчанию): Stop-хук сам UI вопроса не рисует, поэтому эскалация = block-с-инструкцией-позвать-AskUserQuestion; отдельный механизм/событие — out of scope первого прохода. Это наблюдаемый обход отложенного FR-12 — смотрим на ВЫВОД, не на пары spawn↔completion. Где логов/git нет — fail-open (не блок).

## NFR (Non-Functional Requirements)

- **Performance**: синхронно, без сети/subprocess; пропуск линий >1MB; укладывается в Stop-таймаут (<5s).
- **Security**: snippet усечён до 200 символов; per-repo marker/fires scoping через cwd.
- **Reliability**: fail-open везде; atomic marker write; corrupt JSONL-линии пропускаются.
- **Usability**: причина блока простым языком с указанием что прогнать + подсказкой `[UNVERIFIED]`; env kill-switch (`false`).

## Out of Scope

- Улика в ПРЕДЫДУЩЕМ user-ходе (репорт без нового вызова после нового запроса юзера) — known limitation; при необходимости расширить окно по данным fire-лога.
- Семантическая (LLM/embedding) проверка соответствия claim↔tool — гейт чисто лексический/структурный для скорости.
