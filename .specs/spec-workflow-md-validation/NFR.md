# Non-Functional Requirements

## Performance

### NFR-1: Время выполнения хука

Хук ДОЛЖЕН завершаться менее чем за 500ms для типичного проекта 
(до 10 полных фич, до 50 @featureN тегов на фичу).

### NFR-2: Минимальное потребление памяти

Хук НЕ ДОЛЖЕН загружать в память все файлы одновременно.
Использовать стриминговый парсинг для больших файлов.

## Reliability

### NFR-3: Fail-safe выполнение

При любой ошибке хук ДОЛЖЕН:
- Логировать ошибку в ~/.dev-pomogator/logs/specs-validator.log
- Завершаться с exit code 0 (не блокировать промпт)
- НЕ выводить stack trace в stdout

### NFR-4: Устойчивость к битым файлам

Если MD или .feature файл содержит невалидный синтаксис:
- Логировать предупреждение
- Пропустить файл
- Продолжить валидацию остальных

## Usability

### NFR-5: Понятные сообщения

Предупреждения в stdout ДОЛЖНЫ быть:
- На русском языке
- С указанием файла и строки
- С конкретным действием для исправления

### NFR-6: Формат отчёта

validation-report.md ДОЛЖЕН:
- Быть valid markdown
- Содержать timestamp генерации
- Группировать проблемы по типу (NOT_COVERED, ORPHAN, COVERED)

## Security

### NFR-7: Без выполнения стороннего кода

Хук НЕ ДОЛЖЕН:
- Выполнять код из .feature или MD файлов
- Делать сетевые запросы
- Модифицировать файлы кроме validation-report.md

## Compatibility

### NFR-8: Кросс-платформенность

Хук ДОЛЖЕН работать на:
- Windows 10/11
- macOS 12+
- Linux (Ubuntu 20.04+)

### NFR-9: Совместимость с Bun

Хук ДОЛЖЕН использовать только API доступные в Bun 1.0+.
