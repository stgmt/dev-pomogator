# User Stories

## US-1: Автоматическая проверка качества степов

**Как** разработчик использующий BDD,
**Я хочу** чтобы качество моих step definitions автоматически проверялось,
**Чтобы** не допускать пустых или некачественных тестов.

### Acceptance Criteria
- [ ] При остановке агента запускается валидация
- [ ] Определяется язык проекта (TS/Python/C#)
- [ ] Парсятся все step definition файлы
- [ ] Результат выводится в консоль

---

## US-2: Получение детального отчёта

**Как** разработчик,
**Я хочу** получать детальный отчёт о проблемных степах,
**Чтобы** знать какие именно степы нужно исправить.

### Acceptance Criteria
- [ ] Генерируется `steps-validation-report.md`
- [ ] Указывается файл и строка каждого проблемного степа
- [ ] Описывается суть проблемы
- [ ] Степы группируются по статусу (BAD/WARNING/GOOD)

---

## US-3: Понимание что именно не так

**Как** разработчик,
**Я хочу** понимать почему степ считается плохим,
**Чтобы** знать как его исправить.

### Acceptance Criteria
- [ ] Для каждого BAD степа указывается конкретная проблема
- [ ] Примеры проблем: "Only console.log, no assertion", "Empty body", "TODO found"
- [ ] Понятно какой assertion нужно добавить

---

## US-4: Отключение валидации при необходимости

**Как** разработчик,
**Я хочу** иметь возможность отключить валидацию,
**Чтобы** она не мешала когда я сознательно пишу заглушки.

### Acceptance Criteria
- [ ] Создание `.steps-validator.yaml` с `enabled: false` отключает хук
- [ ] Можно указать ignore паттерны для конкретных файлов
- [ ] По умолчанию валидация включена если есть step definitions

---

## US-5: Бесперебойная работа

**Как** разработчик,
**Я хочу** чтобы хук не ломал мой workflow,
**Чтобы** ошибки в парсинге не блокировали работу.

### Acceptance Criteria
- [ ] Ошибки парсинга не выводятся в консоль (только в лог)
- [ ] Хук всегда завершается с кодом 0
- [ ] Проблемные файлы пропускаются
