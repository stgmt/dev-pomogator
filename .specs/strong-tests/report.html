1. TL;DR

💬 Простыми словами Самое главное за 30 секунд. Четыре цифры из настоящих исследований показывают, почему задача вообще существует (LLM пишет много слабых тестов). Пять пунктов — что мы в итоге сделали, чтобы эту проблему чинить.

74.99%
из всех багов, которые тесты от AI не ловят, такая доля — потому что AI забыл подать в тест нужные входные значения (пустую строку, null, очень большое число, юникод). Schäfer arXiv 2406.18181 §3.4.
100% → 4%
конкретный кейс: Llama-3.3 на HumanEval-Java id_81 (date validator) дала 100% покрытие строк и веток — но реально ловила только 4% мутаций. Пропустила Feb 29 leap year boundary. MutGen arXiv 2506.02954 (Jun 2025).
73%
LLM-тестов инженеры Meta приняли в продакшен — но только после того, как AI прогнал свои тесты через mutation-tool и докрутил то, что не ловило баги. Meta ACH FSE 2025.
≥70%
пороговое значение: сколько подломов в коде тест должен ловить, чтобы для критичного кода (auth, billing) считаться достаточным. OutSight + Meta + Levnikolaevich consensus.
📖 Глоссарий — что значат термины выше
Coverage (покрытие)
Сколько строк кода тест выполняет. Не то же самое, что «сколько багов тест ловит». Можно иметь 100% coverage и пропускать большинство багов (см. CORE007_04 ниже и MutGen 2025 пример).
Mutation testing
Тул намеренно ломает код (меняет > на >=, инвертирует if, удаляет строку, и т.д.) и проверяет — упал ли тест. Если тест проходит на сломанном коде — тест бесполезный. Доля пойманных подломов = mutation score / kill rate.
PBT — Property-Based Testing
Вместо «проверь что f(2) === 4» пишешь «проверь что f(x) всегда возвращает положительное при положительном x» — тул сам гоняет 100–1000 случайных входов, включая злые (0, null, MAX_INT, юникод).
Реальные mutation-тулы для каждого стэка
  • TypeScript / JavaScript — Stryker.js (2.9k stars, обновляется ежемесячно). Работает с vitest / jest / mocha / jasmine / karma / cypress.
  • Python — mutmut (1.3k stars, активен). Альтернатива для опытных: cosmic-ray (632 stars).
  • Java — PIT (Pitest, 1.8k stars). Работает с JUnit / TestNG. Платная альтернатива с расширенным набором мутаций: ArcMutate.
  • C# / .NET — Stryker.NET (2.0k stars, обновляется ежемесячно). Работает с xUnit / NUnit / MSTest / Fixie.
  • Rust — cargo-mutants (1.2k stars, обновляется раз в 2 недели). Единственный живой выбор; mutagen и mutpy умерли.
  • Go — 2 живых тула: gremlins для микросервисов (335 stars, но «часы на больших модулях») + avito-tech/go-mutesting для монолитов (форк stale-репы, 669 stars).
Скилл сам определяет стэк по manifests (package.json / pyproject.toml / *.csproj / go.mod / Cargo.toml / pom.xml) и запускает нужный тул.
fast-check / Hypothesis
Реальные тулы для PBT. fast-check — для TS. Hypothesis — для Python. Скилл их подключает в новые тесты.
Shrinking
Когда PBT находит failing input — он его «ужимает» до минимального примера. Бывает integrated (Hypothesis/Hedgehog/fast-check/proptest — лучшее качество) и manual (классика QuickCheck/ScalaCheck — устаревший подход).

3. Конкретные примеры — «слабый» vs «крепкий» тест

Для каждой: показано какую конкретную mutation слабая версия НЕ ловит и как крепкая её ловит. Все примеры — настоящий TypeScript / Python, не abstract foo/bar. Раскрывайте по очереди.

1. PERMISSIVE_MATCHING — toBeDefined() вместо toEqual()

Слабый

import { test, expect } from 'vitest';
import { getUser } from '../src/users';

test('getUser returns a user', async () => {
  const user = await getUser(123);
  expect(user).toBeDefined();
  expect(user).toBeTruthy();
});

Крепкий

test('getUser(123) returns Alice with admin role', async () => {
  const user = await getUser(123);
  expect(user).toEqual({
    id: 123,
    email: 'alice@example.com',
    role: 'admin',
    createdAt: expect.any(Date),
  });
});
Выживает на слабом: Stryker ObjectLiteral mutator на return в getUser — замена объекта на {}. Слабый проходит ({} defined и truthy). Крепкий падает на equality.
2. ASSERTION_ROULETTE — 5 ассертов без сообщений

Слабый

def test_user_profile():
    profile = get_user_profile(user_id=42)
    assert profile is not None
    assert profile.email == "bob@example.com"
    assert profile.role == "user"
    assert profile.last_login_at is not None
    assert profile.subscription.tier == "pro"

Крепкий

def test_user_profile_returns_full_profile_for_id_42():
    profile = get_user_profile(user_id=42)
    assert profile is not None, "expected profile for user 42, got None"
    assert profile.email == "bob@example.com", f"unexpected email: {profile.email}"
    assert profile.role == "user", f"unexpected role: {profile.role}"
    assert profile.last_login_at is not None, "user 42 should have logged in at least once"
    assert profile.subscription.tier == "pro", f"unexpected tier: {profile.subscription.tier}"
Выживает на слабом: mutmut string mutator на "user" → "XX". Тест падает на 3-й строке с пустым AssertionError — developer не знает что упало. С messages — мгновенный triage. Эмпирика: Aljedaani arXiv 2104.14640 — Assertion Roulette в 54% generated JUnit и >50% OSS Android.
3. MAGIC_NUMBER — захардкоженная константа без комментария

Слабый

test('retirement age for a 1965 birth year', () => {
  expect(computeRetirementAge(1965)).toBe(65);
});

Крепкий

const BIRTH_YEAR_1965 = 1965;
// Per UK State Pension Act 2014, anyone born in 1965 retires at 67
// (not 65 — that was pre-2014 law)
const EXPECTED_RETIREMENT_AGE_FOR_1965_COHORT = 67;

test('cohort born 1965 retires at 67 per UK Pension Act 2014', () => {
  expect(computeRetirementAge(BIRTH_YEAR_1965))
    .toBe(EXPECTED_RETIREMENT_AGE_FOR_1965_COHORT);
});
Выживает на слабом: Stryker NumberLiteral mutator на return 65 → return 64. Слабый падает, но reviewer не знает 65 правильно или legacy. Крепкий цитирует regulation — можно сверить со статутом. Реальный UK pension policy инцидент 2016.
4. HAPPY_PATH_ONLY — только удачный сценарий (тест на 29 февраля выявляет баг)

Слабый

def test_parse_iso_timestamp():
    result = parse_iso_timestamp("2026-05-11T10:00:00Z")
    assert result.year == 2026
    assert result.month == 5
    assert result.day == 11

Крепкий — example + boundary + PBT

import pytest
from hypothesis import given, strategies as st

def test_parse_iso_timestamp_rejects_feb_29_non_leap():
    with pytest.raises(ValueError, match="invalid date"):
        parse_iso_timestamp("2025-02-29T00:00:00Z")  # 2025 not leap

def test_parse_iso_timestamp_accepts_feb_29_leap():
    result = parse_iso_timestamp("2024-02-29T00:00:00Z")
    assert result.day == 29

@given(st.datetimes(min_value=datetime(1900, 1, 1),
                    max_value=datetime(2100, 12, 31)))
def test_parse_iso_timestamp_roundtrip(dt):
    iso = dt.replace(tzinfo=timezone.utc).isoformat().replace("+00:00", "Z")
    assert parse_iso_timestamp(iso) == dt.replace(tzinfo=timezone.utc)
Выживает на слабом: mutmut comparison на validator if month > 12: raise → if month >= 12: raise. Слабый никогда не достигает границы. PBT находит off-by-one. Этот же кейс — точно HumanEval-Java id_81 из MutGen 2025.
5. TAUTOLOGICAL — expect(x).toBe(x)

Слабый

test('shouldRetry returns the right boolean', () => {
  const error = new HttpError(503);
  const result = shouldRetry(error);
  // mistake: comparing result to itself
  expect(result).toBe(result);
});

Крепкий

test('shouldRetry returns true for 503 Service Unavailable', () => {
  expect(shouldRetry(new HttpError(503))).toBe(true);
});

test('shouldRetry returns false for 400 Bad Request', () => {
  expect(shouldRetry(new HttpError(400))).toBe(false);
});
Выживает на слабом: Stryker BooleanLiteral mutator на return true → return false. Слабый проходит независимо — x.toBe(x) vacuously true.
6. TRIVIAL_INPUT — реальный CORE007_04 из dev-pomogator

Слабый — историческое

test('logger.info writes structured log entry', () => {
  logger.info('OK');
  expect(captureLog()).toBe('OK');
});

Крепкий

  expect(isDuplicate(a, b)).toBe(true);   // structural equals
  expect(isDuplicate(a, c)).toBe(false);  // currency differs
  expect(isDuplicate(a, d)).toBe(false);  // amount differs
});

10. Что переосмыслили после deep research

💬 Простыми словами Первый research был ОК, но поверхностный. После глубокого прохода нашли 8 уточнений. Для каждого: что было, что стало, и главное — «Значит:» зачем тебе это знать.

Что такое «peer-reviewed» — статья прошла рецензию учёных перед публикацией. Это противоположность блогпосту, который может написать кто угодно. ICSE / FSE / ISSTA / TSE — названия топовых академических конференций и журналов по разработке софта (примерно как «Шампионс лиг» в академмире). arXiv — онлайн-архив научных препринтов.

Группа A: укрепили источники

1. Главная цифра «100% покрытие / 4% mutation» — теперь из научной статьи, не из блога

Было: ссылались на блог-пост OutSight AI (компания, без рецензии).

Стало: ссылаемся на научную статью MutGen (опубликована на arXiv в июне 2025, рецензированная). Там конкретный кейс — модель Llama-3.3 написала тесты для валидатора дат и получила 100% покрытия строк и веток, но реально ловила только 4% подломов. Промах — функция не проверяла что «29 февраля 2025» это невалидная дата (2025 — не високосный год).

Значит: если кто-то скажет «это маркетинговый блог, я не верю» — теперь есть peer-reviewed ссылка с конкретным воспроизводимым кейсом. Аргумент стал научно жёстче.

2. Показали 20-летнюю академическую цепочку, а не одну ссылку

Было: 5 ссылок, часть из которых блоги. Можно было упрекнуть «modная идея AI-эры».

Стало: 8 рецензированных работ за 20 лет — с 2005 (Andrews — «mutation testing валидно как proxy реальных багов») до 2025 (MutGen + Meta ACH). Все опубликованы на топ-конференциях по software engineering: ICSE (International Conference on Software Engineering), FSE (Foundations of SE), ISSTA (Symposium on Software Testing & Analysis), TSE (журнал IEEE Transactions on Software Engineering).

Значит: mutation testing — не хайп 2024-го, а 20-летняя validated техника. Бьёт по аргументу «вы просто на AI-волне».

3. Поправили опечатку в названии конференции (косметика)

Было: у одного из источников (статья 2018 года про то как Google использует mutation testing) была написана конференция «FSE 2018».

Стало: правильно — «ICSE-SEIP 2018». Это разные конференции. SEIP — «Software Engineering in Practice» под-трек ICSE для индустриальных докладов.

Значит: ничего по существу. Если будешь сам цитировать — теперь правильно. Реальный читатель этой разницы не заметит.

Группа B: исправили таблицу инструментов

Было: в спеке упоминался один тул — go-mutesting. Но его основная репа (на zimmski) не обновлялась с 2021 года.

Стало: на самом деле живых тула два: (1) gremlins — современный, активный, но «занимает часы на больших модулях» (для микросервисов ОК); (2) avito-tech/go-mutesting — это форк от Avito от старой репы, поддерживается, более mature catalogue мутаций. Для монолитов лучше.

Значит: если у тебя проект на Go — конкретная рекомендация: gremlins для микросервисов, avito-fork для монолитов. Не наугад.

Группа C: добавили новые design dimensions

6. В таблице PBT-фреймворков добавили колонку Shrinking

Что такое shrinking: когда PBT (property-based testing) находит вход на котором функция падает — тул пытается «ужать» этот вход до минимально-сломанного примера. Например, нашёл что код падает на строке из 200 случайных символов — shrinker сужает до «а на пустой строке тоже падает». Без shrinking — counterexamples нечитаемые.

Было: в таблице фреймворков (fast-check, Hypothesis, jqwik, FsCheck, proptest, gopter) не была показана эта характеристика.

Стало: добавлена колонка с разделением: integrated (тул сам ужимает корректно — Hedgehog, Hypothesis, fast-check, proptest) vs manual (программист пишет shrinker сам — классический QuickCheck, ScalaCheck до 2018, junit-quickcheck).

Значит: при выборе PBT-фреймворка предпочитай тот где integrated. Это напрямую влияет на качество failure messages когда твой тест падает в проде.

7. Diff-scope mutation как default — нужно сделать в v1.1

Что такое diff-scope: запускать mutation testing только на тех файлах которые изменились в текущем коммите/PR, а не на всём репо. Иначе на большом проекте — часы.

Было: run-mutation.ts запускает mutation на полном файле который ему скормили.

Стало: увидели что Google (статья 2018), Stryker.js (--since), cargo-mutants (--in-diff) и PIT (scmMutationCoverage) — все сошлись на этом паттерне как индустриальном default. Пока не сделали — это todo v1.1.

Значит: в v0.1 если запускаешь mutation — single-file mode обязателен (default scope), иначе будет долго. v1.1 — автоматически возьмёт git-дифф.

8. Точная цифра распространённости Assertion Roulette: 54%

Что такое Assertion Roulette: анти-паттерн когда в одном тесте 5+ ассертов без сообщений — когда какой-то падает, неясно какой именно (тест файлит на «3-й строке» и всё).

Было: ссылались на общее исследование которое говорит «Assertion Roulette часто встречается в LLM-тестах».

Стало: нашли точное число — статья Aljedaani 2021: 54% сгенерированных JUnit-тестов содержат Assertion Roulette; >50% в OSS Android-приложениях.

Значит: в FR-2 (audit existing tests) скилл может показывать конкретный %: «вероятность что у тебя есть Assertion Roulette — больше половины». Точно вместо «часто бывает».

13. Behavioural prior (§1.5)

Простыми словами Документ в файле не становится поведенческим prior автоматически. Real-session evidence: тот же агент написал output-invariants-first.md rule и через 2 часа в той же сессии нарушил его — 2 раунда подряд, оба раза с пинком юзера. SKILL.md §1.5 решает это через инлайн загрузку 3 anti-patterns + 2 пинка + closing principle при каждом skill load.

Session-pilot incident summary (2026-05-11)

3 раунда работы над session-pilot dashboard:

  • Раунд 1 (B+) — тесты написаны проактивно потому что юзер явно сформулировал требование. Не показатель внутренней дисциплины.
  • Раунд 2 (D) — добавил Tabulator filter без тестов, доложил «работает» без проверки в браузере. Tabulator silently degraded multiselect → single-select. Пинок #1: «тестов нет нихуя не работает».
  • Раунд 3 (D+) — починил Last Message modal с 6 тестами, доложил «16/16 PASS». Но MSG_04 использовал happy-path fixture (только attachment(text="")), реальные tool-only user/assistant turns не покрыты. Пинок #2: «нажимаю на ласт меседж и нихуя. почему тестов опять нет».

3 anti-patterns A/B/C

  • A. Доложил без проверки в реальности — 16/16 unit PASS, но Tabulator silently degraded в браузере. Урок: перед фразой «работает» — открыть и проверить минимум 1 happy path в реальной среде (browser MCP, debug-screenshot, реальный fixture).
  • B. Happy-path fixture как доказательство покрытия — fixture не включал класс пустых сообщений который реально встречается в JSONL. Урок: fixture должен включать минимум один экземпляр каждого "проблемного" варианта. Реальные данные — основа fixture, не учебная картинка.
  • C. Реактивная дисциплина — пока не пнут — правило output-invariants-first.md написано тем же агентом за 2 часа до того как он его нарушил. Урок: правило применяется к следующему действию агента; skill загружен → действует по умолчанию до конца сессии.

Главный системный вывод

«Знание правила ≠ применение правила. Документ на диске не становится prior автоматически. Skill load обязан активировать behavioural prior на следующее действие агента, не только когда explicitly запрошен mutation testing.»
